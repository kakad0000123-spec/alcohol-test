# 2026-07-09 扁鐵補修欄位（FB25×4.5）

## 做了什麼
- 廠商表單「開孔尺寸」下加選填欄「扁鐵補修(FB25×4.5)」：打算式 `8*2+9+20*2` → 即時顯示長度 mm + 重量 g。
- 共用解析器 `parseFlatbar()`（`src/lib/holes.ts`）＋ upload.html 各一份（純 JS 不 import TS，同步維護）。
- 串全鏈：匯出加「扁鐵補修(mm)/扁鐵重量(g)/扁鐵算式」三欄（周長後）、明細顯示、後台可編輯（含廠商改自己那筆）。
- schema `supabase_phase3.sql`：`hole_uploads` 加 `flatbar_raw`/`flatbar_mm`。已上線實測全過。

## 為什麼
- 廠商反饋：補孔周邊的隔柵扁鐵也要配合修補、有長度，希望跟周長一樣自動算。
- 一孔穿過多支扁鐵、各補一段 → 長度是「多段相加」，故用計算機式算式欄，完全比照他們填 Excel 的習慣。
- 重量＝長度×固定斷面（kg/m = 25×4.5×0.00785 = 0.883125，kg/m 與 g/mm 數值相等）→ 純衍生、即時算不入庫。

## 踩了坑
- 硬相依：DB 沒開欄前，表單 insert 會帶 `flatbar_*` key → PostgREST 整筆拒收。**先跑 migration 才能 push**，用 anon key select 新欄回 200 確認後才部署。
- upload.html 那個「service_role 要 no-store」hook 對廠商前端是誤報（瀏覽器 anon POST，非 Next service_role GET），略過。
- `npm` 非互動 shell 不在 PATH，要 `export PATH=/opt/homebrew/bin`。

## 否決了什麼／為什麼（墓碑）
- **單一數字框（廠商自算總數）**：丟掉分段明細、後台無法核對，也沒發揮「自動算」的意義——反饋要的正是自動算。
- **動態 repeater 一段一列**：手機施工現場填表阻力大、過度設計。
- **重量存 DB 欄**：純衍生值會與長度不同步，改即時算。
- **扁鐵獨立於孔的分支（Excel 007 型：孔無施作卻有扁鐵）**：Po 確認上傳當下不會發生 → 當孔的附屬選填欄，省掉整條分支。
- **扁鐵單獨拍照**：沿用完工照即可（Po 定）。
